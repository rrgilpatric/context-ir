repo-native doc stack. It gives one concrete reviewer-usable case walkthrough

`oracle_signal_smoke_d` is the representative internal case for this source

## Why This Case

`oracle_signal_smoke_d` makes the project thesis less abstract. The current
stack already says Context IR works by analyzing a supported static Python
subset into a `SemanticProgram`, deriving proven dependencies plus explicit
unresolved or unsupported frontier, and compiling budgeted context. This case
is useful because it asks a reviewer to inspect that thesis on a shape that
depends on methods, class relationships, and narrow `@dataclass` facts, which
is why `PLAN.md` and `BUILDLOG.md` treat it as the accepted broader surface in
that direction.

## What The Case Shows

- The semantic-first claim stays concrete when the reviewer looks at a case
  that depends on method, class, inheritance, and narrow dataclass structure
  instead of only file proximity.
- The reviewer can read this case as an example of the repo's current
  conservative behavior: prove what the supported subset can justify, and keep
  unresolved or unsupported frontier explicit rather than fabricating support.
- Because `oracle_signal_smoke_d` is one of the four assets in the accepted
  internal quad matrix, it is a valid illustrative row inside the currently
  allowed fixed-scope comparison that `context_ir` wins all `8/8` task-budget
  rows and leads the provider-average aggregate within that matrix only.
- The case is therefore useful as a reviewer-facing internal example, not as a
  broader claim about external standing, broader Python coverage, or product
  maturity.
